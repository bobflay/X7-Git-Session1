What is Version Control?

Version control is like having a detailed history book for your work, but much more powerful. Think of it as a system that remembers every change you make.

Real-World Analogies:

Google Docs History

Just like how Google Docs shows you every edit made to a document, who made it, and when - but for any type of file.

Save Games

Like having multiple save slots in a video game. You can go back to any previous "save point" if something goes wrong.

Photo Albums

Imagine a photo album that automatically takes a picture every time you change something, with notes about what changed.

Legal Documents

Like how lawyers track changes in contracts - every revision is documented, dated, and can be undone.

What Version Control Tracks:

  • What changed: Exactly which lines, words, or parts were modified
  • When it changed: Precise timestamp of every modification
  • Who changed it: Which person made each change
  • Why it changed: Notes explaining the reason for each change
  • How to undo: Ability to reverse any change instantly

Who Uses Version Control?

  • Writers: Track drafts of books, articles, and manuscripts
  • Designers: Manage versions of logos, layouts, and graphics
  • Students: Keep track of thesis drafts and research papers
  • Legal Teams: Version control for contracts and legal documents
  • Anyone: Who works on files that change over time and needs to collaborate

Benefits for Everyone:

  • Never lose work again
  • See exactly what changed between versions
  • Collaborate without overwriting each other's work
  • Go back to any previous version instantly
  • Work on different ideas simultaneously

Why Git Exists - The Problem It Solves

Before Git and version control systems, developers faced many challenges when working on code projects.

Problems Before Git:

File Chaos

Developers would save files like:
project.js
project_v2.js
project_final.js
project_final_FINAL.js

Lost Work

No backup meant one mistake could delete weeks of work. Hard drives crashed, files got corrupted, and entire projects disappeared.

Collaboration Nightmare

Multiple people working on the same project meant emailing files back and forth, overwriting each other's changes, and losing work.

No Change History

Impossible to see what changed, when it changed, or who changed it. Debugging was like detective work without clues.

Fear of Experimentation

Trying new features meant risking the entire project. Developers were afraid to experiment or refactor code.

Deployment Disasters

No way to know exactly which version was running in production, making rollbacks and updates extremely risky.

How Git Solves These Problems:

  • Complete History: Every change is tracked with who, what, when, and why
  • Safe Experimentation: Create branches to try new ideas without breaking working code
  • True Collaboration: Multiple people can work simultaneously without conflicts
  • Distributed Backups: Every clone is a complete backup of the entire project history
  • Easy Rollbacks: Go back to any previous version instantly
  • Merge Intelligence: Automatically combine changes from different contributors

Real-World Example:

Before Git: "The website is broken! Who changed the login page? When? We need to go back to yesterday's version, but which file is that?"

With Git: git log --oneline shows exactly what changed, git revert abc123 undoes the problematic change in seconds.

Git Workflow: The Three Trees

Git manages your files across three main areas, often called "trees" or states. Understanding this is key to using Git effectively.

The Three Areas:

üå≥ 1. Working Directory

  • What: The files you see and edit.
  • State: Unmodified, Modified, or Untracked.
  • Analog: Your desk where you are currently writing.
  • Command: git status (to see changes)

üì¶ 2. Staging Area (Index)

  • What: A holding area for changes you want to include in the *next* commit.
  • State: Staged.
  • Analog: A box where you put finished items ready to be shipped.
  • Command: git add . (to add changes)

üìú 3. Repository (History)

  • What: The permanent, compressed history of your project.
  • State: Committed.
  • Analog: The library archive where shipped items are permanently stored.
  • Command: git commit (to save staged changes)

The Basic Workflow:

1. Modify

Edit files in the Working Directory

‚Üí

2. Stage

git add . moves changes to the Staging Area

‚Üí

3. Commit

git commit saves the snapshot to the Repository

The git log command is your project's history book. It shows all the commits that have been made to the repository.

  • Combined with --oneline is very powerful

Branching - Working in Isolation

Branches are Git's superpower. They allow you to create an independent line of development to work on new features or fixes without affecting the main, stable code.

Key Concepts:

Main/Master

The default and primary branch. It is usually considered the stable, production-ready code.

Feature Branch

A new branch created to work on a specific feature (e.g., feature/login). It keeps work isolated.

Hotfix Branch

A new branch created to quickly fix a bug in the production code. Merged back quickly.

HEAD

A pointer that always points to the branch you are currently working on (or the last commit you checked out).

Branching Workflow Example:

Imagine you are on the main branch. Your team wants to add a new interactive quiz feature at the end without breaking the existing presentation.

üöÄ Step-by-Step Process:

Step 1: Check current status and see existing branches
$ git status # Check what branch you're on
On branch main nothing to commit, working tree clean
$ git branch # List all branches
* main
Step 2: Create a new branch and switch to it
$ git checkout -b feature/interactive-quiz # Create and switch in one command
Switched to a new branch 'feature/interactive-quiz'
Step 3: Work on the feature (create new files, edit existing ones)
... Edit index.html, styles.css, script.js and add quiz.html ...
Step 4: Check what files changed
$ git status # See what files were modified
On branch feature/interactive-quiz Untracked files: quiz.html Modified files: index.html script.js styles.css
Step 5: Stage and commit your changes
$ git add . # Stage all changes
$ git commit -m "feat: implement interactive quiz feature" # Save the changes
[feature/interactive-quiz 1234567] feat: implement interactive quiz feature 4 files changed, 170 insertions(+)
Step 6: Push the new branch to the remote repository (GitHub)
$ git push -u origin feature/interactive-quiz
* [new branch] feature/interactive-quiz -> feature/interactive-quiz
Step 7: Merge the new branch back into main (Next Slide)

Merging and Collaboration

Merging is the process of combining the history of two branches (e.g., your feature branch and the main branch).

The Merge Workflow:

1. Switch to Main

git checkout main

You always merge *into* the target branch.

2. Pull Latest

git pull origin main

Ensure your main branch is completely up-to-date before merging.

3. Merge

git merge feature/branch-name

Combine the changes from your feature branch into main.

Handling Conflicts:

A merge conflict happens when Git doesn't know which change to keep because two branches modified the same lines in the same file. Git pauses the merge and asks you to fix it manually.

Conflict Resolution Steps:

  1. Git reports conflict: CONFLICT (content): Merge conflict in index.html
  2. Open the conflicting file. You will see markers: 
    <<<<<<< HEAD (main)
<p>Main branch version</p>
========
<p>Feature branch version</p>
>>>>>>> feature/interactive-quiz
  3. Manually edit the file to remove the markers and keep only the correct code.
  4. git add . (to mark the conflict as resolved)
  5. git commit (to create the merge commit and finalize)

Pull Requests (PRs) - The Collaboration Gate

A Pull Request (PR) is a feature of Git hosting platforms (GitHub, GitLab, Bitbucket). It's a formal way to ask other developers to review your work before merging it into a shared, stable branch (like main).

What is a Pull Request?

The Request

You are requesting that code from your feature branch be pulled into the target branch (usually main).

The Review

It opens a web-based interface for the team to view, comment on, and test the changes before merging the branch.

Why Pull Requests Are Important:

  • Code Review: Team members can review your code before it's merged
  • Discussion: Discuss implementation details and suggest improvements
  • Quality Control: Catch bugs, security issues, and maintain code standards
  • Documentation: Create a permanent record of what changed and why
  • Collaboration: Essential for team projects and open source contributions

Real-Life Analogy:

üìù Essay Review

Your draft: Code changes in your branch

Ask for review: Create a pull request

Teacher feedback: Code review comments

Revisions: Update your code based on feedback

Final grade: PR approved and merged

üèóÔ∏è Building Permit

Your plans: Proposed code changes

Submit permit: Open a pull request

Inspector review: Team reviews your code

Make changes: Address review feedback

Approval: Merge into main branch

How Pull Requests Work (GitHub/GitLab/Bitbucket):

  1. Push Branch: You run git push origin feature/name.
  2. Create PR: You go to GitHub/GitLab and click "Create Pull Request" for your new branch.
  3. Review: Teammates review the code, suggest changes, and approve.
  4. Merge: Once approved, the PR is merged into the main branch via the web interface.
  5. Delete: The feature branch is usually deleted after a successful merge.

Rebase vs. Merge (Advanced)

Both git merge and git rebase integrate changes from one branch into another, but they do it in fundamentally different ways, creating different history structures.

Core Differences:

üîÄ Merge

  • Preserves complete history
  • Creates a merge commit
  • Non-destructive operation
  • Shows when branches were merged
  • Can create complex history graph

üìù Rebase

  • Creates linear history
  • No merge commits
  • Rewrites commit history
  • Cleaner, easier to follow
  • Simplifies project timeline

Visual Comparison: Merge vs Rebase

üìç Before: Initial State

A ‚Üí B (main)
C (feature)

üîÄ After Merge

A ‚Üí B ‚Üí M (Merge Commit)
. ‚§¢
C

History is preserved, showing that a parallel branch (C) was merged at point M.

üìù After Rebase (C onto B)

A ‚Üí B ‚Üí C'

History is rewritten to be linear. C is moved and reapplied (C') after B, as if the work was done sequentially.

Rule of Thumb:

When to Use Merge:

  • Integrating a feature into main (final step)
  • Whenever you want to preserve the actual history (when the branch work started)
  • When combining commits that have already been pushed to a shared remote
  • **Rule:** Merge shared branches.

When to Use Rebase:

  • Cleaning up your local commits before pushing
  • Updating your feature branch with the latest main changes (to make merging easier later)
  • To maintain a perfectly linear, easy-to-read history
  • **Rule:** Rebase your private, unpushed branches.

‚ö†Ô∏è The Golden Rule of Git:

Do NOT rebase commits that have already been pushed to a shared remote repository.

Rewriting shared history causes conflicts and confusion for the entire team.

Troubleshooting and Recovery

Don't panic! Git has powerful tools to help you undo almost any mistake.

Mistake #1: Changes not yet committed (in Working Directory/Staging Area)

Git Recovery Commands
Demo
Try It
$ git checkout -- index.html # Discard changes in working directory for ONE file
$ git restore . # Discard ALL un-staged changes in working directory (Newer command)
$ git reset HEAD index.html # Unstage a file (Move from Staging back to Working Directory)
Unstaged changes after reset: M index.html
$

Mistake #2: I committed the changes, but they are only on my computer (Local Repository)

Un-doing Local Commits
Demo
Try It
$ git commit --amend # Fix the most recent commit message or add more files to it
[main 1234567] Fix: Typo in slide title Date: Thu Oct 10 11:00:00 2025 +0300 1 file changed, 1 insertion(+), 1 deletion(-)
$ git reset HEAD~1 # Undo the last commit locally (keeps the file changes in Working Directory)
Unstaged changes after reset: M index.html M styles.css
$ git reset --hard HEAD~2 # WARNING! Hard reset to 2 commits ago (DELETES ALL changes since that point!)
HEAD is now at def456e Update about me slide and styles
$

Mistake #3: I committed the changes and PUSHED them to the remote (Shared History)

This is the trickiest scenario because you are affecting other developers' work. You must use a non-destructive method.

Un-doing Pushed Commits (Safe)
$ git revert abc123 # Create a NEW commit that undoes the changes of the bad commit (abc123)
[main def456] Revert "problematic commit" 1 file changed, 5 deletions(-)
$ git push origin main # Push the new revert commit to the remote
To https://github.com/bobflay/GitSession2_2024.git d4e5f6g..def456 main -> main

‚úÖ Always Revert, Never Force Push!

When dealing with shared, pushed history, git revert is the safe, recommended way to undo work. It preserves history and prevents conflicts.
